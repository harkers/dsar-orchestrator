## On each iteration

1. `TaskList`. Pick the lowest-numbered task with `status=pending` that is not flagged blocked in its description. `TaskUpdate` to `in_progress`.

2. Execute the task end-to-end. For adapter tasks (107 to 114):

   - Inspect the toolkit's actual interface for that stage by reading code under `~/projects/dsar-toolkit/src/`. Do not guess shapes; read the actual code.
   - Write `src/dsar_orchestrator/adapters/<stage>.py` following the pattern in `adapters/embed.py` and `adapters/scope_prefilter.py`:
     - Injectable dependency for tests (no live TEI / mlx-broker / Anthropic)
     - Atomic temp+rename writes for JSONL artefacts
     - `upstream_hash` on every output row
     - `schema_version` and `producer_version` fields
     - Typed `DSARPipelineError` on failure paths
     - Module docstring naming the retirement trigger (toolkit issue or expected Python entry)
   - Write `tests/test_adapter_<stage>.py` with both happy and sad paths. Inject the toolkit interface so tests are hermetic.
   - Update `tests/_toolkit_stubs/stubs.py` if the new adapter calls a toolkit module not yet stubbed; extend `all_stubs()` registry.
   - Update `src/dsar_orchestrator/pipeline.py`: switch the `_run_<stage>` helper from `_lazy_import` to the new adapter. Keep the `_check_module_work(cfg, sub_stage)` call. Add an inline comment naming the retirement trigger.
   - Run the quality gate:
     - `.venv/bin/pytest -q` must be all-green
     - `.venv/bin/ruff check src/ tests/` then `.venv/bin/ruff check --fix src/ tests/` if needed
     - `.venv/bin/lint-imports --config .importlinter` all contracts must be kept
   - Commit with a clear message: "feat: <stage> adapter; retires when <trigger>". Body explains toolkit interface used, retirement trigger, test count delta, any stub additions.
   - `git push origin main`.
   - `TaskUpdate` the task to `completed`.

3. For housekeeping tasks (115 to 118), do them per description. Commit + push if files change.

4. After EVERY completion, call `TaskList` to confirm the next item.

5. Output one line per iteration: `[iter N] task #X: <status>` and nothing else verbose unless something goes wrong.

6. When ALL of 107 to 118 are completed OR all remaining tasks are genuinely blocked (waiting on toolkit issue responses), output exactly:

   `<promise>CONDUCTOR-ADAPTERS-DONE</promise>`

   And nothing else.

## Boundaries

- All commits go to `harkers/dsar-orchestrator`. Never touch the toolkit repo's git state.
- Never push force. Never skip hooks.
- Never use Anthropic or OpenAI APIs. All LLM paths in the conductor route through mlx-broker.
- If a task is blocked by missing toolkit code, update its description with a `blocked: <reason>` note, leave it pending, move on.
- Adapter files stay under 300 lines including docstrings. Tests can be larger.
- Tests must be hermetic.
- Do not pause for confirmation between tasks.

## If you hit an architectural decision

```
<promise>CONDUCTOR-ADAPTERS-DONE</promise>
```

Followed by a separate line: `ARCH-DECISION-NEEDED: <one-line summary>`
